# 20-upgrade: Airgap Environment OpenShift 4.20 → 4.21 Upgrade

## Overview

To upgrade OpenShift in an internet-disconnected airgap environment, three things are required.

| Component | Role |
|-----------|------|
| **oc-mirror** | Mirror images + graph data needed for upgrade to the internal registry |
| **IDMS** (ImageDigestMirrorSet) | Redirects cluster image pull requests to the internal registry |
| **OSUS** (OpenShift Update Service) | Provides local upgrade graph → **displays upgrade button in console** |

```
[Bastion - internet connected]          [Airgap cluster]
  quay.io/openshift-release  →   Internal registry  →  OSUS  →  CVO  →  Console
  (mirrored with oc-mirror)             (IDMS applied)     (graph)  (status)  (upgrade UI)
```

---

## Prerequisites

### Required Tools (Bastion host)

```bash
# Install oc-mirror v2 plugin
# https://mirror.openshift.com/pub/openshift-v4/x86_64/clients/ocp/latest/oc-mirror.tar.gz
tar xzf oc-mirror.tar.gz
chmod +x oc-mirror
mv oc-mirror /usr/local/bin/

# Check version
oc mirror version
```

### Internal Registry Requirements

- TLS certificate configured (self-signed acceptable, but must be added to cluster's trusted CA)
- Sufficient storage space (4.20→4.21 upgrade images: approximately 20~30GB)
- Address accessible from cluster nodes

### Environment Variable Setup (Bastion)

```bash
export REGISTRY_HOST="registry.example.com"   # Internal registry address
export MIRROR_PREFIX="ocp4"                   # Storage path prefix inside registry
export PULL_SECRET="/path/to/pull-secret.json" # Red Hat pull secret
```

## Step 1: Image Mirroring (Bastion → Internal Registry)

### 1-1. Prepare imageset-config.yaml

Use the [imageset-config.yaml](./imageset-config.yaml) file.

```yaml
apiVersion: mirror.openshift.io/v2alpha1
kind: ImageSetConfiguration
mirror:
  platform:
    graph: true          # Also mirror graph data image (used by OSUS)
    channels:
      - name: stable-4.21
        minVersion: 4.20.0
        maxVersion: 4.21.99
        shortestPath: true
```

> `graph: true` must be set.
> Without this option, OSUS cannot provide upgrade paths and the upgrade will not appear in the console.

### 1-2. Run Mirroring

```bash
# Create mirroring working directory
mkdir -p /mnt/mirror

# Run oc-mirror (internet connection required)
oc mirror \
  --config=imageset-config.yaml \
  --workspace file:///mnt/mirror \
  docker://${REGISTRY_HOST}/${MIRROR_PREFIX} \
  --v2

# Check output after completion
ls /mnt/mirror/working-dir/mirror-to-disk/
```

> Upon completion, the following files are generated under `/mnt/mirror/working-dir/cluster-resources/`:
> - `idms-oc-mirror.yaml` — ImageDigestMirrorSet
> - `itms-oc-mirror.yaml` — ImageTagMirrorSet
> - `updateService.yaml` — UpdateService CR (includes graphDataImage path)
> - `release-signatures/` — Release signatures

### 1-3. Check graphDataImage Path

```bash
# Check the actual image path in the UpdateService file generated by oc-mirror
cat /mnt/mirror/working-dir/cluster-resources/updateService.yaml
```

Example output:
```yaml
spec:
  releases: registry.example.com/ocp4/openshift/release-images
  graphDataImage: registry.example.com/ocp4/openshift/graph-image:latest
```

> Use this value when replacing `REGISTRY_HOST` in `update-service.yaml`.

---

## Step 2: Apply IDMS to Cluster

```bash
# Apply IDMS generated by oc-mirror
oc apply -f /mnt/mirror/working-dir/cluster-resources/idms-oc-mirror.yaml
oc apply -f /mnt/mirror/working-dir/cluster-resources/itms-oc-mirror.yaml

# Apply release signatures
oc apply -f /mnt/mirror/working-dir/cluster-resources/release-signatures/

# Wait for MachineConfig rollout (IDMS is applied without node restart, but verify)
oc get mcp
```

Verify IDMS application:
```bash
oc get imagedigestmirrorset
oc get imagetagmirrorset
```

---

## Step 3: Deploy OpenShift Update Service (OSUS)

For the upgrade button to appear in the Console, OSUS must be running inside the cluster
to provide the upgrade graph.

### 3-1. Install OSUS Operator

> **Airgap environment note:** The `redhat-operators` CatalogSource must be mirrored to the internal registry.
> If not mirrored, refer to [3-1-b Manual Installation](#3-1-b-manual-installation-without-olm).

```bash
# Apply Namespace + OperatorGroup + Subscription
oc apply -f update-service.yaml --field-manager=apply-server
# (Apply only the Namespace/OperatorGroup/Subscription sections at the top of update-service.yaml first)

# Verify CSV installation
oc get csv -n openshift-update-service
# NAME                        DISPLAY                        VERSION   PHASE
# update-service.v4.x.x      OpenShift Update Service       4.x.x     Succeeded
```

#### 3-1-b. Manual Installation (without OLM)

If OLM catalog mirroring is not available, apply OSUS manifests directly:

```bash
# Download OSUS operator manifests (from bastion with internet connection)
oc adm release extract \
  --from=registry.example.com/ocp4/openshift/release-images:4.21.0 \
  --to=/tmp/release-manifests \
  --credentials-requests

# Or obtain directly from GitHub
# https://github.com/openshift/cincinnati-operator
```

### 3-2. Apply UpdateService CR

Use the `updateService.yaml` generated by oc-mirror, or replace `REGISTRY_HOST`
in `update-service.yaml` and apply.

```bash
# Replace REGISTRY_HOST and apply
sed 's|REGISTRY_HOST|registry.example.com|g' update-service.yaml | \
  oc apply -f - -n openshift-update-service

# Or use the oc-mirror generated file
oc apply -f /mnt/mirror/working-dir/cluster-resources/updateService.yaml
```

Verify OSUS Pod startup:
```bash
oc get pods -n openshift-update-service
# NAME                       READY   STATUS    RESTARTS   AGE
# service-7d9f8c6b4-xxxxx    1/1     Running   0          2m
# service-7d9f8c6b4-yyyyy    1/1     Running   0          2m
```

Verify OSUS Route:
```bash
oc get route -n openshift-update-service
# NAME      HOST/PORT                                             ...
# service   service-openshift-update-service.apps.cluster.com    ...
```

## Step 4: Configure ClusterVersion Upstream

Configure the CVO (Cluster Version Operator) to retrieve the upgrade graph from local OSUS.

```bash
# Get OSUS Route address
OSUS_HOST=$(oc get route service \
  -n openshift-update-service \
  -o jsonpath='{.status.ingress[0].host}')

echo "OSUS endpoint: https://${OSUS_HOST}/graph"

# Change ClusterVersion upstream
oc patch clusterversion version \
  --type=merge \
  -p "{\"spec\":{\"upstream\":\"https://${OSUS_HOST}/graph\"}}"

# Configure channel (stable-4.21)
oc patch clusterversion version \
  --type=merge \
  -p '{"spec":{"channel":"stable-4.21"}}'
```

Verify configuration:
```bash
oc get clusterversion version -o yaml | grep -A5 'spec:'
# spec:
#   channel: stable-4.21
#   upstream: https://service-openshift-update-service.apps.cluster.com/graph
```

CVO takes 1-2 minutes to detect the upgrade path:
```bash
oc get clusterversion version
# NAME      VERSION   AVAILABLE   PROGRESSING   SINCE   STATUS
# version   4.20.x    True        False         ...     4.21.x available
```

---

## Step 5: Verify Upgrade in Console

1. Access OpenShift Console
2. Navigate to **Administration → Cluster Settings**
3. Click the **Update** tab
4. Confirm that version `4.21.x` appears in channel `stable-4.21`

When the upgrade button appears, click it to start the upgrade, or
run via CLI:

```bash
# Check available upgrade versions
oc adm upgrade

# Upgrade to a specific version (recommended)
oc adm upgrade --to=4.21.x

# Direct upgrade with image digest (for forcing versions outside channel)
OCP_VERSION="4.21.0"
DIGEST=$(oc get clusterversion version \
  -o jsonpath="{.status.availableUpdates[?(@.version==\"${OCP_VERSION}\")].image}")
oc adm upgrade --to-image="${DIGEST}" --allow-explicit-upgrade
```

## Upgrade Progress Monitoring

```bash
# Overall progress
oc get clusterversion version

# Detailed events
oc describe clusterversion version

# Upgrade status per operator
oc get clusteroperators

# Filter operators still upgrading
oc get clusteroperators \
  -o custom-columns=\
'NAME:.metadata.name,VERSION:.status.versions[0].version,AVAILABLE:.status.conditions[?(@.type=="Available")].status,PROGRESSING:.status.conditions[?(@.type=="Progressing")].status'

# Node upgrade status (MachineConfigPool)
oc get mcp

# Real-time logs
oc logs -n openshift-cluster-version \
  -l app=cluster-version-operator -f
```

## Troubleshooting

### When Upgrade Does Not Appear in Console

```bash
# 1. Check OSUS graph endpoint directly
OSUS_HOST=$(oc get route service -n openshift-update-service \
  -o jsonpath='{.status.ingress[0].host}')

# Check cluster_id and channel
CLUSTER_ID=$(oc get clusterversion version -o jsonpath='{.spec.clusterID}')
curl -sH "Accept: application/json" \
  "https://${OSUS_HOST}/graph?channel=stable-4.21&id=${CLUSTER_ID}" | \
  python3 -m json.tool 2>/dev/null || \
  echo "(JSON parsing failed - check raw response)"

# 2. Verify CVO can reach the OSUS endpoint
oc get clusterversion version -o jsonpath='{.spec.upstream}'

# 3. Check CVO logs for upgrade detection errors
oc logs -n openshift-cluster-version \
  -l app=cluster-version-operator --tail=50 | grep -i "error\|upstream\|graph"

# 4. Check for graphDataImage pull errors
oc describe pod -n openshift-update-service | grep -A5 "Events:"
```

### Image Pull Failure After Applying IDMS

```bash
# Check IDMS contents
oc get imagedigestmirrorset -o yaml

# Add registry CA trust
oc create configmap registry-ca \
  --from-file=registry.example.com=/path/to/ca.crt \
  -n openshift-config

oc patch image.config.openshift.io/cluster \
  --type=merge \
  -p '{"spec":{"additionalTrustedCA":{"name":"registry-ca"}}}'
```

### Interrupt/Retry Upgrade

```bash
# Force re-check upgrade status
oc adm upgrade --force
```
